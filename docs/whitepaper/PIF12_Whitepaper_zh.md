## 摘要

**PIF12**（Pay It Forward — 12-Year Legacy，把善傳下去．十二年傳承）是一場為期十二年、部署在 **Ethereum** 上的鏈上信任與感恩實驗。連續十二年，每一年都會發行一款不可轉讓的 token——以日本的御守（omamori）為形——標記一個不斷生長、彼此願意為對方挺身而出的人的圓圈。

PIF12 也是對我們這個時代一個安靜難題的刻意回應：科技讓我們連結了所有人，卻不知怎地讓我們更孤獨了。當 AI 讓「不需要另一個人也能運作」變得前所未有地容易，PIF12 偏偏反其道而行——用同一套鏈上工具，不是把人從迴圈中移除，而是把人重新織回彼此之中。

**哪些是承諾，哪些是實驗。** 這裡有兩件事是堅定的承諾：其一，連續十二年，每一年發行一款 token；其二，每年一場我都會親自舉辦的**實體聚會**——可能兩三個人，也可能兩三百人，想想就覺得有趣！其餘大部分——尤其是後面會講到的具體遊戲機制（§5–§6）——都是實驗，可能會變：它取決於的，不只是我自己的餘裕，還有 AI 能替我分擔多少，以及我們究竟有多想繼續一起玩。

PIF12 v1.0.0 做了一個決定性的設計選擇：**沒有分數。** 留下的，是人與人連結最簡單的一份紀錄：

- **你幫過幾個「不同的」人**，以及
- **幾個「不同的」人幫過你。**

只數*人*，不數點數——甚至連事件也不數——這抹去了排行榜、抹去了可被操弄的空間，只留下真正重要的事：*你是否真的在某個人需要時出現在他身邊，以及是否有人為你挺身而出。*

圍繞著這個核心，安靜地運轉著五套系統：**靈魂綁定**身分、聯邦式的 **Curator Network（信任星座）**、**雙方共同確認**的互助、三模式的**社會恢復**，以及**免 gas**的參與方式——讓任何人都不需要懂錢包，就能收到一份禮物。這五套系統全都寫進已部署的合約裡；它們分期啟動——第一年（2026）是領取年，Curator Network 的星星層自第二年（2027）開始開啟（見 §14）。

> **北極星。** *每一個善行都是一顆星。我們一起，畫出一個將照亮黑暗的星座。* PIF12 是一顆**北極星，而非太陽**——它給的是**方向**，不是熱度。它本來就該小巧、記錄感恩，並帶著一點點樂趣。若有夥伴加入、星座因此生長，那就更好了。

## 1. 哲學：「把善傳下去」究竟是什麼意思

把善傳下去（Pay-It-Forward）不是一種經濟、也不是一張計分板。它是一場連鎖反應：

1. 有人幫了你，不求回報。
2. 你逐漸累積出一種被社群「接住」的真切感受。
3. 這份經驗推動你去幫助另一個人。
4. 鏈條延續下去——不是以交易的形式，而是以漣漪的形式。

因此，一份忠實的鏈上紀錄應該：

- **數人**，因為連結是以一張張臉孔來衡量，而非數字；
- **不把給予變成競賽**，因為給予該是發自內心的；
- **保持簡單**，因為它一旦變成遊戲，就不再是禮物了。

PIF12 v1.0.0 是這個專案至今對此最誠實的表達：**記錄感恩，保持簡單，帶點樂趣。**

## 2. 御守

多年來，Jason 一直守著一個安靜的儀式——在農曆新年前後，向他在乎的人送出祝福。PIF12 把這個儀式搬上鏈。

形式借自日本的御守——廟宇裡那種裝著一句手寫祈願的小護身符。PIF12 不具宗教性質；它只借用這份心意的精神：某種小小的、刻意的、*屬於你*的東西。每一年，會以靈魂綁定代幣的形式鑄造一款 token，送給願意走過這趟十二年旅程的人。你買不到它。你也轉讓不了它。**它唯一的價值，是你真的來了。**

第一個圓圈，啟動了這個星座——對每一個與我們生命有過交集的人敞開，數量不設上限。從第二年（2027）起，隨著**小太陽（Lightkeeper）**將這份精神傳承下去，每一位都會點亮另一顆**星星（Star）**，星座因此持續生長。

我也加上一個屬於我自己的承諾。每一年，年復一年，連續十二年，我都會辦一場實體聚會，把這個圓圈聚在一起——而我，會在場。其他人來不來是另一回事；重點始終是「出現」。一個建立在「你真的來了」之上的身分，至少值得每年有一個真的可以前來的地方。

## 3. 架構一覽

`PIF12Nexus` 是單一一份 Solidity 合約，把五套系統縫合進一個 ERC-1155 裡。

| 層 | 選擇 | 為什麼 |
| :--- | :--- | :--- |
| **鏈** | Ethereum L1 | 十二年的永久性（見 §10） |
| **代幣** | ERC-1155，靈魂綁定 | 多代幣身分；不可轉讓 |
| **可升級性** | UUPS proxy | 十二年的實驗必須能夠演進 |
| **免 gas** | ERC-2771 + Paymaster | 目標：讓成員避開 gas（盡力而為；第一年用 relay 領取，第二年起改用 Paymaster meta-tx） |
| **授權** | GPL-3.0-or-later | Copyleft：把開放傳下去（§9） |
| **依賴套件** | OpenZeppelin Upgradeable v5 | 經審計的標準原語 |

Token id 編號方案：

- **id 1–12** — 生肖年 token（每年共用一個 id；供給量等於該年入列的小太陽人數）。id 1 = 馬年（2026）⋯⋯ id 12 = 蛇年（2037）。
- **id 100 起** — 個人的「星星」token，每一顆都是唯一 id、供給量為 1（自第二年、2027 起首次發行）。

## 4. Curator Network（小太陽 → 星星）

PIF12 v1.0.0 以一個聯邦式的 **Curator Network（信任星座）**取代了原本的單一發行者模型：一圈核心成員，每個人又可以再邀請自己的人。

完整的網路已寫進已部署的合約，但它是分期展開的：第一年（2026，馬年）是領取年——只有第一層，向不限人數的領取者敞開；第二層的星星發行自第二年（2027，羊年）啟動。

```
            發起人 Jason（Initiator）
                     │  鑄造一款生肖年 token
                     ▼
      第一層 — 小太陽（Lightkeeper）   ← 核心成員，那一年的圓圈
                     │  發一顆星星給他們敬佩的人
                     ▼
      第二層 — 星星（Star）            ← 一位小太陽點亮的人
                     ⊘ 兩層止步 — 星星不能再往下發
```

- **發起人**鑄造一款生肖年 token 給一位**小太陽**（合約中的中性詞是 *Curator*）。這透過營運用的 relay 完成，所以收受者從不簽署任何交易——他們只是領取。
- **每位小太陽**——自第二年（2027）發行開關打開後——可以免 gas 地發出最多 **50 顆星星**給他們敬佩的人，附上一張客製化的圖與一段簡短的感謝話語（這段話語存在事件日誌裡，從不進儲存——前端就是審核層）。
- **網路止於兩層。** 一顆星星不會變成小太陽；它不能再往下發。信任刻意只向外輻射一跳。
- **有時限，但有寬限保護。** 一位小太陽只能在其生肖年的窗口內被*入列*，但其發星星的權利會一直開放到 `max(該年截止日, 入列日 + 100 天)`——所以即使是在一年快結束時才入列的人，也仍有充裕的時間去點亮屬於自己的星星。一個誠實的後果是：因為星星發行在整個第一年都被閘門關著（見 §14），這個窗口實務上是自第二年（2027）的隊伍起才適用。第一年的成員以馬年 token 持有者的身分參與；**星星發行從第二年的隊伍開始。**
- **柔性排除，不沒收。** 若有成員偏離了精神，合約管理者（早期是創辦人的錢包；自第三年起改為多簽——見 §12）可以呼叫 `revokeCurator`——封鎖其*未來*的發行，同時讓他既有的 token 與已經點亮的星星原封不動。沒有任何 token 會被沒收。更常見的是，排除只是自然發生的：社群不再伸出援手，於是 `peopleHelpedBy` 安靜地維持在低點。不會有人被貼上恥辱的標籤。

一個單一的 `issuanceEnabled` 旗標把關第二層發行——部署即關閉，並維持關閉直到整個第一年結束（2026：小太陽領取），由合約管理者在第二年（2027）星星體驗正式上線時打開。一個早已寫好的功能，靠一個旗標來啟動，而不是靠一次有風險的升級。

**透明度：發起人本身也是一位小太陽。** Jason 持有一款年度 token，並以圓圈中一名成員的身分參與——他是起點，而非高人一等的階層。

## 5. 核心指標：唯一人數計算

這是 v1.0.0 的核心，也是這個專案至今最乾淨的一刻。

> **關於哪些是實驗性的。** 本節與下一節（§6）是 PIF12 的*遊戲化機制*——也是最可能演變的部分。不同於「十二年發行」與「每年實體聚會」，這些機制並不是承諾；UUPS 可升級性（§3）的存在，正是為了讓它們能隨著我自己的餘裕、AI 能替我分擔多少，以及社群想一起玩的意願而調整。

每位成員的檔案就是兩個數字：

```
peopleHelped[a]    — `a` 幫過幾個「不同的」人
peopleHelpedBy[a]  — 幾個「不同的」人幫過 `a`
```

就這樣。沒有價值、沒有上限、沒有冷卻時間、沒有分數。

**為什麼數人，不數事件。** 幫同一個人一百次，仍然只算**一次**。要讓計數器往上跳，你需要*另一個不同的人*參與進來。因此，要灌水就得招募許多不同的、彼此同意的成員——於是操弄的成本會隨著一個真實社交網路的規模而放大，而不是隨著交易筆數。抗女巫（Sybil resistance）就這麼從設計裡自然落出來，不費分文。

**為什麼 `peopleHelpedBy` 才是更真實的訊號。** `peopleHelped` 是你能控制的——你隨時可以去幫別人。`peopleHelpedBy` 則是由*別人*授予的：它只有在不同的人*選擇*幫你、而你們雙方共同確認時才會上升。它代表社群整體的判斷：到底選擇擁抱了誰，這是你自己宣稱不來的。這使它成為地位最自然的量度——而且，溫和地說，也是最自然的納入機制：一位與精神不合拍的成員，自然就累積不到它。

## 6. 互助：雙方共同確認

合約怎麼*知道*一個人幫了另一個人？它需要兩個人一起。

```
步驟 1   給予者   → recordHelp(recipient)        // 建立一筆待確認的紀錄；此刻尚不計數
步驟 2   受助者   → confirmHelp(id, memo)         // 受助者確認，可選擇附上一句道謝
```

在某一組（給予者、受助者）配對的**第一次**確認幫助時——而且只有第一次——`peopleHelped[giver]` 與 `peopleHelpedBy[recipient]` 各自加一。同一組配對之間重複的幫助是受歡迎的（你可以留下一段新的話），但永遠不會重複計數。

| 攻擊 | 防禦 |
| :--- | :--- |
| A 單方面替偏愛的 B 灌數字 | B 必須確認——記錄 ≠ 計數 |
| B 捏造自己受過幫助 | A 必須先記錄——沒有給予者就沒有宣稱 |
| 鏈下脅迫（「確認，不然……」） | 30 天窗口；B 大可永遠不確認，紀錄自行過期 |
| 女巫叢集 | 成員必須持有 SBT 才能參與；數*人*把一個叢集的上限卡在它自身的規模，而封閉的互助圖在鏈上一覽無遺 |

可選的 `memo` 是一段自由文字的道謝，只發送到事件日誌——從不上鏈儲存、從不由合約驗證。前端決定要呈現什麼；鏈上日誌內容的設計，本就是永久且公開的。

## 7. 社會恢復——三種模式，預設由管理者守護

弄丟一個錢包，不該意味著失去十二年的身分。PIF12 提供三種姿態；預設那種不需要任何設定。

- **AdminDefault（預設）。** 發起人的管理錢包——早期是創辦人的單一錢包，自第三年起升級為多簽（見 §12）——可以把一位成員的整個身分遷移到一個新錢包。零設定——你開箱即受保護。
- **Guardian（守護人，可退出）。** 指定兩位守護人（guardian），兩位都必須已是網路成員。一位發起，*另一位*在一段時間鎖（48 小時–30 天，對近期活躍的錢包會延長）後執行。一個短暫被入侵的錢包，無法立刻就把「管理者協助恢復」這條後路給封死，因為更改你的恢復模式本身就有時間鎖。
- **Lone Wolf（孤狼模式，可退出）。** 沒有人能恢復你。最大限度的自主主權；最大限度的風險。

恢復會遷移一切使你*之所以是你*的東西：token（透過 burn-and-mint，繞過靈魂綁定鎖）、兩個人數計數器、小太陽的身分與配額，以及你的恢復設定。兩個錢包都有 180 天的冷卻期，為任何濫用劃下界線。

三種模式全都寫進已部署的合約，從第一天起就生效；預設模式零設定自動套用。稍後才推出的，只是*切換*模式的自助介面——在它推出前，任何想改成 Guardian 或孤狼模式的成員，都可以直接請創辦人幫忙修改。

## 8. 免 gas——這是目標，不是承諾

任何成員都不該*需要*懂錢包、持有 ETH，或簽署一筆看不懂的交易，才能收到一份禮物。

- **領取一款 token**（第一年、2026 起上線）走 relay：收受者登入（例如透過內嵌錢包服務商），由營運用的 relay 鑄造到他們的位址。他們在鏈上不簽署任何東西。
- **發出一顆星星**（第二年、2027 起）是一筆 meta-transaction：小太陽簽署 typed data，由 relayer 提交，再由 **Paymaster** 的預算支付 gas。

合約透過 `_msgSender()`（ERC-2771）解析出真正的行為者，所以一個受信任的 forwarder 可以代任何人 relay。

這是**目標，不是保證。** 完全免 gas 的體驗取決於網路不壅塞、以及 Paymaster 預算撐得住。比起承諾「在任何情況下都零 gas」，這個專案寧可少承諾、多交付。

## 9. 授權：以 Copyleft 把善傳下去

PIF12 v1.0.0 採用 **GPL-3.0-or-later** 授權——這是相對於早先寬鬆授權計畫的一次刻意轉向。

理由本身就扣著主題。一張寬鬆授權（MIT）說的是「拿去吧，隨你怎麼用，封閉它也行。」一張 copyleft 授權說的是「**拿去吧，在它之上去建造，但替下一個人保持它的開放。**」第二句話，正是把「把善傳下去」的精神寫進了法律條文裡。對一個整個身分都關乎互惠慷慨的專案來說，授權本身就是宣言的一部分。

合約納入了採 MIT 授權的 OpenZeppelin 元件；MIT 對 GPL 是單向相容的，原始的授權聲明都予以保留（見 `NOTICE`）。本白皮書的文字採 CC BY-SA 4.0——散文部分也照樣分享（share-alike），同一份精神用在文字上。

## 10. 為什麼選 Ethereum L1

PIF12 是一份**十二年**的承諾，而它所棲身的那條鏈，是它極少數近乎**不可逆**的決定之一：合約可以升級，但若不重新部署並遷移每一枚 token，它就無法被搬走。

這個單一事實主宰了選擇。一個整套論證都建立在*永久性*之上的專案，必須建立在現有最永久的土壤上。Layer-2 網路提供更便宜的 gas 與更多當下的關注，但每一條都是年輕的系統，在十二年的尺度上各自背著自己的排序器（sequencer）、金庫與治理風險。今年最熱的鏈，三年後可能就冷了。

所以 PIF12 為**存活而非熱度**最佳化——可信的中立性，以及在 2037 年仍然還在的最高機率。一個低頻身分專案的 gas 成本並不高（一圈開放的成員，每人一年頂多幾次鏈上動作），由營運用的 relay、以及自第二年起的 Paymaster 吸收。而到頭來，這個敘事也更強：*為了十二年，承諾在最永久的鏈上* 說的，比 *哪裡有關注就部署在哪裡* 要多得多。

這就是把北極星用在基礎設施上：選方向，不選熱度。

PIF12 v1.0.0 出貨時帶著**內部測試與安全審查**。它尚未經過外部審查——而且這個專案在所有它發聲的地方都把這點講得清清楚楚。

- **48 個 Foundry 測試**，全數通過——涵蓋部署/初始化、角色分離、鑄造、靈魂綁定鎖、暫停、三種恢復模式、發行閘門、星星發行、互助計數，以及 UUPS 升級授權。
- 用 Slither 做**靜態分析**。
- 過程中浮現並修正了一個真實的發現：合約超出了 **EIP-170** 的 24 KB bytecode 上限約 2 KB；把約 60 個 `require` 字串改成 custom errors，讓它舒舒服服地降到上限之下，還留有餘裕。

這個姿態與第一年刻意收窄的介面相稱：只有 relay 鑄造與靈魂綁定代幣，星星發行的程式碼整年被閘門關著。**一次外部安全審查明訂為第二年（2027）星星發行啟用之前的前提**——這是前提條件，不是事後補件。在那次審查完成之前，每一份關於 PIF12 的對外陳述，都把它描述為*經過安全審查（security-reviewed）*，而非*經過審計（audited）*——一個這個專案刻意守住的區別。

## 12. 治理的演進（十二年的弧線）

PIF12 以滾動式三年視野規劃：下面的第 1–3 年是已承諾的；後面的階段是預期的方向，每一年重新確認（見 §14）。

- **第 1–2 年（創世 — 2026 馬年 / 2027 羊年）。** Jason 擔任發起人；管理 / 升級 / 暫停權由創辦人的單一錢包持有（一個刻意輕量的起步——已在 §13 揭露）；一個收窄的營運角色負責鑄造。
- **第 3 年（2028，猴年）起（成長）。** 隨著核心成員浮現，管理權自創辦人的單一錢包遷移至一個納入社群成員的多簽——這是一次角色交接（`grantRole`，接著 `renounceRole`），不需要修改合約。參數變更移交給鏈下治理（一人一票——任何 SBT 持有者）。
- **第 6–8 年（成熟）。** Jason 朝見證者的角色退去；升級權移交給社群。
- **第 9–10 年（自治）。** 升級權被鎖定或棄置；合約趨近不可變。
- **第 11–12 年（傳承）。** 最後一款 token（蛇年）被鑄造；專案告一段落；合約作為一份公共紀錄留存下來。

一個已記下的未來方向，*不*屬於 v1.0.0：到了第 5–6 年，網路或許會打開，讓更多成員能自己入列其他人——許多顆小太陽，各有自己的軌道。那是對兩層模型的一次重大改動，也正是 UUPS 可升級性的用武之地。它*刻意*不寫進 v1；把第 5 年的事過度工程到第 1 年，只會徒增複雜度與攻擊面。

## 13. 風險與緩解

**地位競爭。** *成員開始互相比計數。* — 沒有價值、沒有上限落差、沒有排行榜；指標是「不同的人」，讀起來像連結，不像名次。第一年立下的社群規範，會勸阻把它當成炫耀的本錢。

**女巫叢集。** *一圈成員互相「幫助」以灌大計數。* — 數人把一個圈子的上限卡在它自身的規模；參與需要 SBT；封閉的互助圖可見、可審；`revokeCurator` 封鎖壞行為者的發行。

**脅迫。** *「確認我幫過你，不然……」* — 受助者大可永遠不確認；紀錄在 30 天後過期；這次嘗試留下的公開紀錄，本身就帶著聲譽上的代價。

**鏈的存續。** *這條鏈在 2037 年還會在嗎？* — Ethereum L1 是現有機率最高的賭注；合約註明分叉或遷移在理論上仍屬可能，但並不在計畫之內。

**管理權的中心化。** *管理者可以遷移任何預設模式的成員——而在第 1–2 年，管理者是單一的創辦人錢包，還不是多簽。* — 在第 1–2 年，合約管理（升級、暫停、預設錢包恢復）由單一的創辦人錢包持有——一個刻意輕量的起步，在這裡先講明，而不是讓人事後才發現。那一把鑰匙能執行 UUPS 升級與即時的預設模式恢復；它存放在一個硬體錢包上，並有離線備份，弄丟它會被當成像被偷一樣嚴重看待。隨著核心成員浮現（目標：第 3 年，2028），管理權遷移至一個社群多簽；不需要修改合約。在那之前，任何成員都可以隨時退出創辦人預設恢復、改為 Guardian 或孤狼模式，把那份權柄收回到自己手上。

## 14. 路線圖（滾動式三年視野）

接下來三年是已承諾的；之外的一切都是方向，每年重新確認。

- **第一年（2026，馬年）— 開放小太陽領取（目標：2026 年 6 月 21 日）。** 把 `PIF12Nexus` 部署到 Ethereum；任何與我們生命有過交集的人，都能領取一款馬年 token——數量不設上限。這一年的工作，是免 gas 的領取前端與 landing page。星星發行整年維持關閉。
- **第二年（2027，羊年）— 星星發行。** 兩層結構啟動：串接免 gas 的 meta-transaction 與 Paymaster；在合約管理者把 `issuanceEnabled` 打開之前，完成一次**外部安全審查**；小太陽開始點亮星星（Star，pSBT）。
- **第三年（2028，猴年）— 核心成員。** 核心成員浮現；管理權自創辦人的單一錢包遷移至社群多簽——這是一次角色交接，不是合約改動。
- **第 4–12 年（方向性）** — 每年一款生肖 token；路線圖以滾動式三年視野確認，所以較後面的里程碑維持方向性：治理逐步去中心化，專案最終孕育出 **Republic Labs（共和實驗室）**——拉丁文 res publica（眾人之事）：一個自給自足的社群，成員在其中發起屬於自己的小實驗。到那時，也許會冒出無數個新的 PIF12——一整片小實驗的星座，逐一亮起。

## 北極星

> *每一個善行都是一顆星。我們一起，畫出一個將照亮黑暗的星座。*

設計越早期，它越想去*衡量*善意。v1.0.0 停止衡量，只是單純地*記錄*它——誰為了誰挺身而出。合約守住那份紀錄；人，守住那份意義。

北極星不發熱。它指路。
